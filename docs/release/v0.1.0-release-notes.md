# CORTEX v0.1.0

v0.1.0 is the first production-shaped release candidate for the CORTEX log
management platform.

## Highlights

- Multi-service Spring Boot ring: gateway, ingest, processor, remediation,
  indexer, monitoring, echo, and Eureka.
- Kafka-backed ingest and anomaly pipeline with DLQ surfaces.
- Postgres-backed raw-log and anomaly read models.
- Spring AI anomaly classification with deterministic local WireMock smoke.
- Loki/Quickwit fan-out surfaces and Quickwit admin/search integration.
- Fix-first remediation pipeline with audit outcomes and guarded human fallback.
- Docker compose, Helm, Terraform, and Ansible deployment scaffolding.
- Grafana dashboards, Prometheus SLO burn-rate rules, and SLO catalog.
- Full CI/CD pipeline: root `mvn verify` with Testcontainers, an eight-image
  Docker build matrix, a blocking Trivy image scan, GHCR publish, and keyless
  cosign signing on trusted push/tag events.
- Newman API tests in CI: the six per-service Postman collections run against
  the booted compose stack on every PR and merge.
- Gatling SLO load test: a `GatewayLoadSimulation` asserts the gateway
  round-trip latency and success-rate budgets in CI.
- Release-prep tooling for SBOM generation, cosign image signing, and GitHub
  Release creation.

## Verification Before Publishing

The release owner must attach the exact verification evidence from the final
release branch:

- `mvn verify`
- P10/P11/P12/P13/P17 smoke gates
- P18 release-prep smoke
- Newman suites that apply to the release target
- SBOM generated from the release commit
- Cosign signatures for every published image

## Known Limitations

- Azure Service Bus is provisioned by Terraform as the intended production
  broker, but the implemented service runtime remains Kafka-wired until the
  later binder/runtime migration.
- The actual GitHub Release and image signing are gated by explicit operator
  approval; the repository contains tooling and runbooks, not a published
  release artifact by default.
- A small set of HIGH/CRITICAL image CVEs is framework-locked to the Spring
  Boot 3.3.x line (spring-core, spring-security-web, spring-boot, kafka-clients,
  and its transitive lz4-java) and is allowlisted in `.trivyignore` with a
  per-CVE justification; clearing them requires the Spring Boot 3.4+/Spring
  Kafka 3.3+ migration tracked in issue #156. The Trivy gate remains blocking
  for every new fixable CVE.
